# Performant N-point FFT Kernel

## Context and goal

The hand-coded `vec-fft8.c` (N=8, all six test geometries) passes. We now want
a kernel that works for arbitrary power-of-two N and runs well on the target
hardware. The broader purpose is to use vector FFT kernels to stress-test the
architecture, measure what compiled C can achieve, and drive compiler/backend
improvements where needed.

The file being built is `python/zamlet/kernel_tests/fft/vec-fftN.c`. It is
parameterised by `-DFFT_N=<N>` and `-DTWIDDLE_HEADER=...` (generated by
`gen_twiddles.py`). The `vec-fft8.c` file is kept as a reference.

## Why a naive port does not work

The obvious design — "load contiguous A and B halves, complex-multiply with a
twiddle vector, butterfly, store" — breaks down in two places:

1. **Stages with pair-distance smaller than `vl`.** In radix-2 DIT, stage s has
   pair-distance `d = 2^s`. For `d < vl`, each butterfly uses an A and a B
   element that are only `d` apart in memory. To fill a vl-length A vector we
   must gather every `d`-th element, which is a stride-`d` memory compress:
   elements move through up to `vl/2` lanes in the physical VPU mesh. For
   `vl=1024` that is a half-chip data move per load. Prohibitively expensive.

2. **Stages with pair-distance at or above `vl`.** Here each butterfly's A and
   B sides are each contiguous runs of `vl` elements, so unit-stride loads
   work. But if the whole FFT uses this structure, we pay a full memory
   round-trip per stage — `log₂N` round-trips for the whole transform.

The design has to handle regime 1 without long-distance mesh moves, and should
do several stages per memory round-trip so regime 2 is not round-trip-bound.

## Physical model

The VPU is a mesh of lanes. For a vector register of length `vl`, element `i`
sits at lane `i`. Moving data between lanes has cost roughly proportional to
mesh distance. `vslide1up`/`vslide1down` (±1-lane moves) and stage-aligned
gathers with short reach are cheap; long-reach gathers (stride-k compress with
large k) are expensive.

Consequence: any permutation in the hot loop must be expressible as a gather
from a compact index set, not a long-reach compress.

## VL and R selection

Two parameters are chosen per run before any FFT work begins:

- `vl = min(hw_vl_max, N/2)`. The chunk structure processes data in
  register pairs (two vl-length vectors = 2·vl elements per pair), so
  at minimum we need `N ≥ 2·vl`, i.e. `vl ≤ N/2`. At exactly `vl = N/2`
  we get R=2, one register pair, chunk_size = N.
- `R = min(8, N/vl)`, the number of complex registers per chunk. Since
  `N/vl ≥ 2` is always a power of 2, `R ∈ {2, 4, 8}`.

Consequences:

- `chunk_size = R · vl = min(8·vl, N)`.
- Regime A runs `log₂(vl)` stages; Regime B runs `log₂(R)` stages (at
  most 3).
- `R < 8` iff `chunk_size = N`, i.e. the whole FFT fits in one chunk
  and Regime C is skipped.
- `R = 8` iff `chunk_size = 8·vl`. Regime C fires iff additionally
  `8·vl < N`. When it does fire, the super-chunk register count is
  always 8.

## Design overview

The kernel is structured as a radix-(R·vl) FFT with a radix-R outer
structure (R ∈ {2,4,8} per above). We load a chunk of `chunk_size = R ·
vl` complex elements into R complex value registers (2R physical
register groups at `e64,m1`), perform `log₂(chunk_size)` consecutive
FFT stages entirely in the register file, and store the chunk back.
For `N ≤ chunk_size` (equivalently, `R < 8` or exactly `R·vl = N`) the
whole FFT runs from one chunk load. For `N > chunk_size` (only
possible with `R = 8`), the remaining `log₂N − log₂(chunk_size)` stages
are done at coarser scales using an 8-register inter-chunk butterfly
structure, three stages per memory round-trip.

Within a chunk, stages split into two regimes based on pair-distance:

- **Regime A — within-register gather** (`d < vl`): the pair elements live
  inside the same vl-length register. We form the A-side and B-side vectors
  for the stage's butterflies with a pair of vector slides + masked merges
  (no `vrgather`), do one element-wise complex butterfly, then slide+merge
  the results back into the chunk's register layout. See the Regime A
  section below for the group-wise layout that makes this cheap.
- **Regime B — inter-register butterfly** (`d ≥ vl`, up to `d = chunk_size/2`):
  the A side and B side live in different registers. No permutation is
  needed at all. The chunk's registers pair up as `(R_a, R_b)` with `R_b =
  R_a + d/vl`, and we do an element-wise complex butterfly between the two
  registers.
- **Regime C — inter-chunk butterfly** (`D ≥ chunk_size`): runs when
  `N > chunk_size`. Load 8 `vl`-length registers from 8 different chunks at
  the same register-position-within-chunk; this forms an 8-element
  super-chunk. Run the Regime B butterfly on the super-chunk — three stages
  with pair-distances `D, 2D, 4D` for base distance `D = chunk_size` on the
  first pass. For `N > 8·chunk_size`, recurse with `D ← 8·D` on the next
  pass. Three stages per memory round-trip.

Within a chunk, Regime A covers stages `0..log₂(vl) − 1` (log₂(vl)
stages) and Regime B covers the next log₂(R) stages (pair-distances
`vl, 2·vl, ..., (R/2)·vl`). Regime C (R = 8 only) covers the remaining
`log₂(N/chunk_size)` stages, grouped three at a time into
`⌈log₂(N/chunk_size) / 3⌉` passes. The `3` at the inter-chunk scale
comes from `log₂(8) = 3` for the 8-register super-chunk butterfly.

## Register budget (e64,m1)

At `e64,m1` there are 32 vector register groups. The complex FFT needs
both a real and imaginary vector per logical value, so `R` complex
values consume `2R` physical register groups. With `R ≤ 8` we use at
most 16 groups for chunk data, leaving at least 16 for twiddles, gather
indices, masks, and scratch. Going to m2 halves the number of register
groups, so `R = 8` would consume the whole file. Hence m1.

## Precomputed seed blocks

All twiddle vectors in the kernel are built by unit-stride `vle64` of
rows from a single header-defined array, optionally followed by scalar
multiplication. `gen_twiddles.py` emits

```c
#define SEED_BLOCK_K <effective_K>
static double seed_block_re[TWIDDLE_LOG2N][SEED_BLOCK_K]
    __attribute__((section(".data.vpu64")));
static double seed_block_im[TWIDDLE_LOG2N][SEED_BLOCK_K]
    __attribute__((section(".data.vpu64")));
```

with `seed_block[j][k] = omega[j]^k` for `j ∈ [0, log₂N)`,
`k ∈ [0, SEED_BLOCK_K)`. `omega[j]` is the existing table of successive
squarings of `ω_N`; every ratio used anywhere in the kernel (all
Regimes) is one of these `omega[j]` values, so one row per `j` covers
everything.

`K` is a CLI parameter to `gen_twiddles.py`, defaulting to 128 and
capped to `min(K, N)`. The effective value is reflected in
`#define SEED_BLOCK_K`. Storage is `log₂N · K · 16` bytes of VPU
memory — e.g., 20 KB at `N = 1024`, `K = 128`.

**Invariant.** Every per-stage twiddle vector in the kernel is derived
from a unit-stride `vle64` of one `seed_block[j]` row. The per-regime
transforms are:

- **Regime A**: tile-replicate the loaded block to length `vl` at init;
  the hot loop uses the stored `W[s]` directly with no scalar multiply.
- **Regime B**: use the loaded block (length `vl`) as `seed[s]`; the hot
  loop forms each pair's twiddle as `base_tw[s][p] × seed[s]`.
- **Regime C**: same as Regime B, indexed by pass and sub-stage.

**Common `build_seed(dest, j, len)` helper.** Loads
`min(len, SEED_BLOCK_K)` entries of `seed_block[j]` via `vle64` into a
vector register. If `len ≤ SEED_BLOCK_K`, done. Otherwise extend to
`len` by repeated doubling, where at each step the current length
doubles from `cur` to `2·cur`:

1. Pick the scalar `s = omega[j + log₂(cur)]` directly from the
   header. This works because `omega[j] = ω_N^(2^j)` implies
   `omega[j]^(2^m) = omega[j+m]`, so `ω^cur = omega[j + log₂(cur)]` for
   any power-of-2 `cur`. No scalar arithmetic — just an `fld`.
2. Complex scalar-vector multiply `tmp = V × s` at length `cur`
   (2 × `vfmul.vf` + `vfsub.vv` / `vfadd.vv`). `tmp` now holds
   `[ω^cur, ω^(cur+1), …, ω^(2·cur−1)]`.
3. `vsetvli 2·cur`, tail-undisturbed, then `vslideup.vx V, tmp, cur`.
   The low `cur` lanes of V are preserved (tail-undisturbed); the high
   `cur` lanes receive `tmp`. V now holds
   `[ω^0, …, ω^(2·cur−1)]` at length `2·cur`.

Doublings are `log₂(len / SEED_BLOCK_K)` and in practice zero for the
typical `vl ≤ 128` geometries.

## Regime A — within-register gather

**Setup.** Within a chunk, registers `V_0..V_{R−1}` hold consecutive
vl-length sub-chunks of the buffer: `V_i` holds buffer positions
`i·vl .. (i+1)·vl − 1` (both real and imag, tracked as parallel registers
`V_i_re`, `V_i_im`). For regime A the pair-distance `d < vl`, so every
butterfly pair lives inside a single `V_i`, and each `V_i` holds `vl/2`
butterflies worth of data.

**Process in pairs `(V_{2p}, V_{2p+1})`.** Within a pair, we build one
vl-length `H0` and one vl-length `H1` carrying all vl butterflies for that
pair, arranged so lane `j` of `H0` pairs with lane `j` of `H1`. The butterfly
is then one element-wise complex vec op:

```
tmp   = W · H1        // complex vec-vec multiply, W = stage-s twiddle vector
H0'   = H0 + tmp
H1'   = H0 − tmp
```

`W` is the same vl-length vector for every pair (and every chunk), since the
twiddle depends only on lane position within a group, not on which `V_i` the
group came from.

**Group-wise lane assignment (the key trick).** For stage `s` with
pair-distance `d = 2^s` and group size `m = 2d`, divide H0's `vl` lanes into
`vl/(2d)` groups of `2d` contiguous lanes. Within group `g`:

- H0 lanes `g·2d .. g·2d + d − 1` hold V_a's A-lanes of group `g` (which sit
  at those same positions of V_a).
- H0 lanes `g·2d + d .. g·2d + 2d − 1` hold V_b's A-lanes of group `g`
  (which sit at positions `g·2d .. g·2d + d − 1` of V_b — a uniform `+d`
  shift).

H1 mirrors H0 with B-lanes instead of A-lanes: first `d` lanes of each group
from V_a's B-lanes (at `g·2d + d .. g·2d + 2d − 1` of V_a — a uniform `−d`
shift), next `d` from V_b's B-lanes (at those same positions of V_b —
identity).

**Why this is cheap.** The V_a contribution to H0 and the V_b contribution
to H1 are masked identity copies: the elements are already at their target
lanes, so the mesh sends no inter-lane messages for that half of the gather
traffic. The V_b contribution to H0 and the V_a contribution to H1 are
single uniform slides by `d`: `vslideup(V_b, d)` and `vslidedown(V_a, d)`.
At stage 0 (`d=1`) this degenerates to the alternating pattern
`[a0, b0, a2, b2, …]` — a `vslide1up`/`vslide1down`. Max per-element
movement is `d`, same as a lane-interleave scheme, but with fewer ops per
stage and with half the elements not moving at all.

**Per-stage cost per pair.** Build H0: `vslideup(V_b, d)` + `vmerge(V_a,
slid_Vb, low_half_mask[s])`. Build H1: `vslidedown(V_a, d)` + `vmerge(slid_Va,
V_b, low_half_mask[s])`. Butterfly: 4 `vfmul` + 1 `vfsub` + 1 `vfadd`
(complex multiply `W·H1`) + 2 `vfadd`/`vfsub` (final combine). Pack back
V_a': `vslideup(H1', d)` + `vmerge(H0', slid_H1', low_half_mask[s])`. Pack
back V_b': `vslidedown(H0', d)` + `vmerge(slid_H0', H1', low_half_mask[s])`.
`R/2` pairs per chunk. The slide distance is `d = 2^s` in lanes; at
`s = 0` this is `vslide1up`/`vslide1down`. A single per-stage mask
(`low_half_mask[s]` = 1 when `lane mod 2d < d`) is reused across all
four gather/pack steps within a pair.

**Worked example: vl=4, stage 0 (d=1).** Pair `(V_a, V_b)` with
`V_a = [a0,a1,a2,a3]`, `V_b = [b0,b1,b2,b3]`. Four butterflies across the
pair: `(a0,a1), (a2,a3)` in V_a and `(b0,b1), (b2,b3)` in V_b.

```
H0 = [a0, b0, a2, b2]                  // A-sides; groups of 2d=2 alternate V_a/V_b
H1 = [a1, b1, a3, b3]                  // B-sides
W  = [1, 1, 1, 1]                      // stage-0 twiddle = 1
```

H0 is built by `vslideup(V_b, 1)` + `vmerge(V_a, slid_Vb, [1,0,1,0])`
(mask 1 = pick V_a). H1 is `vslidedown(V_a, 1)` + `vmerge(slid_Va, V_b,
[1,0,1,0])`. After butterfly, pack back:

```
V_a' = vmerge(H0', vslideup(H1', 1), [1,0,1,0])
    = [H0'[0], H1'[0], H0'[2], H1'[2]]
V_b' = vmerge(vslidedown(H0', 1), H1', [1,0,1,0])
    = [H0'[1], H1'[1], H0'[3], H1'[3]]
```

V_a's A-elements and V_b's B-elements stay put; V_b's A-elements and V_a's
B-elements slide by 1.

**Worked example: vl=4, stage 1 (d=2).** Same pair. Butterflies
`(a0,a2), (a1,a3)` in V_a and `(b0,b2), (b1,b3)` in V_b. Single group spans
all four lanes.

```
H0 = [a0, a1, b0, b1]     // V_a's A-lanes, then V_b's A-lanes
H1 = [a2, a3, b2, b3]     // V_a's B-lanes, then V_b's B-lanes
W  = [1, −i, 1, −i]       // W_4^q where q = lane mod d
```

H0 = `vmerge(V_a, vslideup(V_b, 2), [1,1,0,0])`. H1 = `vmerge(vslidedown(V_a,
2), V_b, [1,1,0,0])`. V_a's A-side (lanes 0, 1) and V_b's B-side (lanes 2, 3)
don't move; the other halves slide by `d = 2`. Pack back:

```
V_a' = vmerge(H0', vslideup(H1', 2), [1,1,0,0])
    = [H0'[0], H0'[1], H1'[0], H1'[1]]
V_b' = vmerge(vslidedown(H0', 2), H1', [1,1,0,0])
    = [H0'[2], H0'[3], H1'[2], H1'[3]]
```

**Tables emitted by `init_tables()` for regime A, per stage s:**

- `W_re[s]`, `W_im[s]` — vl-length twiddle vectors, built once at init
  and stored in `.data.vpu64`. Pattern: within each group of `2d` lanes,
  `W[q] = W_{2d}^{q mod d}` (i.e. the geometric progression
  `[1, ω, ω², …, ω^{d−1}]` repeated for the V_a half and the V_b half of
  each group; `ω = W_{2d}^1 = omega[log₂N − s − 1]`). Built by:
  1. `build_seed(tmp, log₂N − s − 1, d)` loads the length-`d` core
     `[1, ω, …, ω^{d−1}]` into a vector register.
  2. Tile-replicate to length `vl` via `log₂(vl/d)` `vslideup` steps
     with tail-undisturbed: at each step, `vsetvli 2·cur`,
     `vslideup.vx V, V, cur`. Each step doubles the populated prefix by
     copying it to the newly-enabled lanes; the existing lanes are
     preserved by tail-undisturbed semantics.
  3. `vse64` to `.data.vpu64`.

`low_half_mask[s]` (1 where `lane mod 2d < d`) is not stored. It is
rebuilt each regime-A stage in three instructions: `vid.v` + `vand.vx
idx, 2d−1` + `vmslt.vx mask, idx, d`.

The slide distance `d = 2^s` is a compile-time or stage-loop-constant
integer, not a table. All tables shared across the `R/2` pairs per chunk.

## Regime B — inter-register butterfly

**Setup.** After regime A finishes, the pair-distance has reached `d = vl`.
From here up to `d = (R/2)·vl` (`log₂(R)` stages) the two sides of every
butterfly live in different chunk registers. No `vrgather` is needed; each
stage is an element-wise complex butterfly between two registers `(R_a, R_b)`.

**Register pairing per stage.** Let regime-B stage `s ∈ {0, ..., log₂(R) − 1}`
have pair-distance `d_s = vl · 2^s`. The `R` chunk registers
`V_0..V_{R−1}` form `R/2` register pairs at every stage, with
`R_b_index = R_a_index + 2^s`. Shown for `R = 8`:

| stage s | d_s   | register pairs                                  |
|---------|-------|-------------------------------------------------|
| 0       | vl    | (V0,V1), (V2,V3), (V4,V5), (V6,V7)              |
| 1       | 2·vl  | (V0,V2), (V1,V3), (V4,V6), (V5,V7)              |
| 2       | 4·vl  | (V0,V4), (V1,V5), (V2,V6), (V3,V7)              |

For `R = 4` only stages 0 and 1 run, with pairs
`{(V0,V1), (V2,V3)}` and `{(V0,V2), (V1,V3)}`. For `R = 2` only stage 0
runs, with the single pair `(V0, V1)`.

**Twiddle structure.** For stage `s`, the butterfly at buffer-group position
`j ∈ [0, d_s)` uses twiddle `W_{2d_s}^{j} = exp(−2πi · j / (2·d_s))`. A
register pair `(R_a, R_b)` carries `vl` consecutive butterflies whose
positions share a common offset `p · vl` (for some `p ∈ [0, d_s/vl)`); within
the pair they span `j = p·vl + k`, `k = 0..vl−1`. So the vl-length twiddle
vector for a given pair factors:

```
W_pair(s, p) = W_{2d_s}^{p·vl} · [W_{2d_s}^0, W_{2d_s}^1, …, W_{2d_s}^{vl−1}]
             =   base_tw[s][p]  ·  seed[s]
```

- `seed[s]` — vl-length vector, shared across all pairs at stage `s`.
- `base_tw[s][p]` — complex scalar, one per `p` (i.e. per distinct row of
  register pairs within a stage's group). `p` ranges over `[0, d_s/vl) =
  [0, 2^s)`, so stage `s` has `2^s` scalars (`1, 2, 4, ...`). Total
  across Regime B: `Σ_{s=0}^{log₂R − 1} 2^s = R − 1` complex scalars (1,
  3, or 7 for R = 2, 4, 8).

Mapping from pair index to `p` follows the standard DIT layout: within each
group of `2·d_s/vl` registers, the first `d_s/vl` pairs (`R_a` indices from
the group's low half) take `p = 0, 1, …, d_s/vl − 1` in order.

**Butterfly per register pair.** One complex vec-scalar multiply to form the
pair's twiddle vector; one complex vec-vec multiply + vec add/sub to apply
the butterfly:

```
W = base_tw[s][p] · seed[s]             // complex vec-scalar mul, one per pair
tmp = W · R_b                            // complex vec-vec mul, one per pair
R_a' = R_a + tmp                         // vec add (re & im)
R_b' = R_a − tmp                         // vec sub (re & im)
```

Using the available widening/narrowing ops lets the scalar twiddle be held in
FP registers without spilling.

**Seed construction.** `seed[s]` is the geometric progression with ratio
`ω_s = W_{2d_s}^1 = omega[log₂N − log₂(vl) − s − 1]`. It is built at init
by `build_seed(seed[s], log₂N − log₂(vl) − s − 1, vl)` — unit-stride
`vle64` of the appropriate `seed_block[j]` row, followed by doubling if
`vl > SEED_BLOCK_K`. `init_tables()` runs this once per Regime B stage
and stores each `seed[s]` in `.data.vpu64` for the hot-loop `vle64`.

## Regime C — inter-chunk butterfly

Only present when `N > chunk_size`, which (given `R = min(8, N/vl)`)
happens exactly when `R = 8` and `8·vl < N`. The inter-chunk
super-chunk therefore always uses 8 registers — `R < 8` configurations
skip Regime C entirely. Each pass processes three consecutive FFT stages
(pair-distances `D, 2D, 4D` for some base distance `D ≥ chunk_size`) in
one memory round-trip.

**Setup per pass.** Let pass `P ∈ [0, P_max)` have base pair-distance
`D_P = chunk_size · 8^P`. The stages covered by pass `P` are the three FFT
stages with pair-distances `D_P`, `2·D_P`, `4·D_P`. Total passes:
`P_max = ⌈log₂(N/chunk_size) / 3⌉`.

**Inner loop.** For each pass `P`:

1. Iterate over chunk-group starting addresses `G ∈ {0, 8·D_P, 16·D_P, …,
   N − 8·D_P}` (each group spans `8·D_P` complex elements).
2. For each register position `r ∈ [0, 8)` within a chunk:
   - Load 8 `vl`-length registers from the 8 chunks at offsets `G, G + D_P,
     G + 2·D_P, …, G + 7·D_P`, each at position `r·vl` within its chunk.
     All 8 loads are unit-stride `vl`-length loads.
   - Run Regime B's 3-stage butterfly on `V_0..V_7` (same pairing table,
     same `base_tw × seed` twiddle factoring). The twiddle seeds and base
     scalars for this pass come from Regime C's pass-specific tables (see
     `init_tables()` below).
   - Store the 8 registers back to their original addresses.

Register budget: 16 physical register groups for `V_0..V_7` (re + im),
matching the intra-chunk budget. Twiddle seeds (2 regs) and scratch fit in
the remaining 16 groups.

**Twiddle factoring.** For pass `P`, Regime B sub-stage `s_rel ∈ {0,1,2}`
corresponds to FFT pair-distance `D_P · 2^{s_rel}` and group size
`2·D_P·2^{s_rel}`. A butterfly at buffer position `p` within its group `g`
has `j = p − g·2·D_P·2^{s_rel} ∈ [0, D_P·2^{s_rel})`. Writing `j = p_lin·vl
+ k` (`k ∈ [0, vl)`), the twiddle is

```
W_{2·D_P·2^{s_rel}}^j = base_tw_C[P][s_rel][p_lin] · seed_C[P][s_rel][k]
```

where `seed_C[P][s_rel]` is a `vl`-length vector and
`base_tw_C[P][s_rel]` is an array of `D_P·2^{s_rel}/vl = 8^P · 2^{s_rel}`
complex scalars. (For `P = 0` this reduces to `2^{s_rel}` scalars —
identical to Regime B.) `p_lin` is the linear position of the butterfly's
A-side register within the group, determined by `(r, chunk-index-within-
group)` and the super-chunk pairing.

The seed vectors stay the same length across all passes (`vl`). Only the
base-scalar arrays grow with `P`, which is fine because they live in
scalar memory.

Each `seed_C[P][s_rel]` is built at init via
`build_seed(c_seed[P][s_rel], log₂N − log₂(vl) − 4 − 3P − s_rel, vl)` —
same mechanism as Regime B, with the `omega[]` index shifted further
down per pass and sub-stage.

**Partial final pass.** If `log₂(N/chunk_size)` is not a multiple of 3,
the last pass uses a smaller super-chunk (2 or 4 registers) and performs
only 1 or 2 Regime-B sub-stages. Structurally the same algorithm, just
with a smaller working set. This is only invoked when `N/chunk_size ∈ {2,
4, 16, 32, 128, …}`.

## `init_tables()` layout

Called once at startup after `vsetvli` determines `vl`. Populates:

**Regime A, per stage `s ∈ [0, log₂(vl))`** — all in `.data.vpu64` (vectors
loaded via `vle64`):

- `W_re[s][0..vl−1]`, `W_im[s][0..vl−1]` — stage twiddle vector, built
  from `seed_block[log₂N − s − 1]` at init via load + tile-replicate
  (see Regime A section). `low_half_mask[s]` is not stored; it is
  rebuilt each stage from `vid.v` + `vand.vx` + `vmslt.vx`.

**Regime B, per stage `s ∈ [0, log₂(R))`** (at most 3 stages):

- `seed_re[s][0..vl−1]`, `seed_im[s][0..vl−1]` — in `.data.vpu64`, built
  from `seed_block[log₂N − log₂(vl) − s − 1]` via `build_seed`.
- `base_tw_re[s][0..(2^s)−1]`, `base_tw_im[s][0..(2^s)−1]` — in scalar
  memory (no section attribute), accessed by `fld` + broadcast via
  `vf*.vf`.

**Regime C, per pass `P ∈ [0, P_max)` and sub-stage `s_rel ∈ [0, 3)`**
(only if `N > chunk_size`):

- `c_seed_re[P][s_rel][0..vl−1]`, `c_seed_im[P][s_rel][0..vl−1]` — in
  `.data.vpu64`, built from
  `seed_block[log₂N − log₂(vl) − 4 − 3P − s_rel]` via `build_seed`. One
  `vl`-length seed per sub-stage per pass.
- `c_base_tw_re[P][s_rel][0..(8^P · 2^{s_rel})−1]`,
  `c_base_tw_im[P][s_rel][0..(8^P · 2^{s_rel})−1]` — in scalar memory.
  Base scalars grow with pass number; for the first pass (`P = 0`) there
  are only `1 + 2 + 4 = 7` scalars total, matching Regime B at R = 8.

**Bit-reverse indices** — as in `vec-fft8.c`, produced by `compute_indices()`
in the bitreverse helper.

**Sizing.** All vl-length vector buffers are statically dimensioned by
`MAX_VLMAX` (compile-time upper bound, passed via `-DMAX_VLMAX=…`). The
number of regime-A stages is at most `log₂(MAX_VLMAX)`, bounding the table
array sizes.

**Table construction.** All twiddle-vector construction reduces to two
primitives:

- `build_seed(dest, j, len)` — unit-stride `vle64` of
  `min(len, SEED_BLOCK_K)` entries of `seed_block[j]`; if
  `len > SEED_BLOCK_K`, extend to `len` by repeated doubling. Each
  doubling fetches the scalar `ω^cur = omega[j + log₂(cur)]` from the
  header, complex scalar-vector multiplies the current length-`cur`
  vector to obtain the next `cur` powers, and splices via
  `vslideup.vx` tail-undisturbed. Used as-is for Regime B `seed[s]` and
  Regime C `c_seed[P][s_rel]`.
- `build_regime_a_seed(dest, s)` — `build_seed(tmp, log₂N − s − 1, d)`
  (with `d = 2^s`), then tile-replicate to `vl` by `log₂(vl/d)` doubling
  `vslideup` steps with tail-undisturbed.

Scalar tables (`base_tw`, `c_base_tw`) are geometric progressions in
scalar memory, filled with a short scalar loop at init. Regime B totals
`R − 1` scalars; Regime C scales with pass number but arrays stay short
relative to N.

## Kernel driver structure

Pseudocode for `main()` (mirrors `vec-fft8.c`):

```
main() {
    init input buffer (data_re/im = { k + 0i : k = 0..N-1 });
    compute_indices(N, vl, br_read_idx32, br_write_idx32, /*reverse=*/1);
    widen 32→64 into br_read_idx / br_write_idx;
    init_tables();

    bitreverse_reorder64(N, data, scratch, br_read_idx, br_write_idx);

    for chunk_base in 0, chunk_size, 2·chunk_size, … , N − chunk_size:
        load_chunk(V_0..V_{R−1}, data + chunk_base);    // 2R vle64
        for s in 0 .. log₂(vl) − 1:                     // regime A
            regime_a_stage(V_0..V_{R−1}, s);
        for s in 0 .. log₂(R) − 1:                      // regime B
            regime_b_stage(V_0..V_{R−1}, s);
        store_chunk(V_0..V_{R−1}, data + chunk_base);   // 2R vse64

    for P in 0 .. P_max − 1:                            // regime C
        D_P = chunk_size · 8^P;
        for G in 0, 8·D_P, 16·D_P, …, N − 8·D_P:
            for r in 0 .. 7:
                load V_k (k=0..7) from data + G + k·D_P + r·vl;
                run_regime_b(V_0..V_7,
                             seed = c_seed[P], base_tw = c_base_tw[P]);
                store V_k back;

    verify against expected_re[], expected_im[] within TOL;
}
```

`regime_a_stage` and `regime_b_stage` are inlined (or `static inline`) so
that `V_0..V_{R−1}` remain in fixed register slots throughout the chunk.

## Worked example: N = 32, vl = 4 (R = 8)

With `hw_vl_max ≤ 4`, we get `vl = 4`, `R = min(8, 32/4) = 8`,
`chunk_size = R·vl = 32 = N`, so the whole FFT fits in a single chunk.
Zero Regime C passes. `log₂(chunk_size) = 5 = log₂(vl) + log₂(R) = 2 +
3`, so regime A runs stages 0 and 1 (pair-distance 1 and 2), regime B
runs stages 0, 1, 2 (pair-distance 4, 8, 16).

**After bit-reversed load:**

```
V0 = x[0..3]    V1 = x[4..7]     V2 = x[8..11]   V3 = x[12..15]
V4 = x[16..19]  V5 = x[20..23]   V6 = x[24..27]  V7 = x[28..31]
```

**Regime A stage 0 (d=1)**, **Regime A stage 1 (d=2)** — as in the two
worked examples above, applied to each of the four register pairs.

**Regime B stage 0 (d=4, 2d=8):** pairs `(V0,V1), (V2,V3), (V4,V5), (V6,V7)`.
All pairs have `p = 0`, so `base_tw[0] = [1]`, and every pair's twiddle
vector is simply `seed[0] = [1, e^{−iπ/4}, −i, e^{−i3π/4}]`. Butterfly each
pair.

**Regime B stage 1 (d=8, 2d=16):** pairs `(V0,V2), (V1,V3), (V4,V6),
(V5,V7)`. `p ∈ {0, 1}` with `base_tw[1] = [1, −i]` (since `W_{16}^4 = W_4^1
= −i`). The four pairs map `(V0,V2) → p=0`, `(V1,V3) → p=1`, `(V4,V6) →
p=0`, `(V5,V7) → p=1`. `seed[1] = [1, e^{−iπ/8}, e^{−i2π/8}, e^{−i3π/8}]`.

**Regime B stage 2 (d=16, 2d=32):** pairs `(V0,V4), (V1,V5), (V2,V6),
(V3,V7)`. `p = 0, 1, 2, 3` respectively, with `base_tw[2] = [W_{32}^0,
W_{32}^4, W_{32}^8, W_{32}^{12}] = [1, e^{−iπ/4}, −i, e^{−i3π/4}]`.
`seed[2] = [W_{32}^0, W_{32}^1, W_{32}^2, W_{32}^3]`.

Store `V0..V7` back to `data` and compare against `expected_re[],
expected_im[]`.

## Worked example: N = 32, vl = 16 (R = 2)

With `hw_vl_max ≥ 16`, we cap `vl = min(hw_vl_max, N/2) = 16`, giving
`R = min(8, 32/16) = 2`, `chunk_size = 32 = N`, one chunk, no Regime C.
`log₂(chunk_size) = 5 = log₂(vl) + log₂(R) = 4 + 1`, so Regime A runs
stages 0..3 (pair-distance 1, 2, 4, 8), Regime B runs a single stage
(pair-distance 16) on the single pair `(V0, V1)`.

**After bit-reversed load:** `V0 = x[0..15]`, `V1 = x[16..31]`.

**Regime A stages 0..3** — applied to the single pair `(V0, V1)`.
Stage `s` uses `W = tile-replicate([1, ω, ..., ω^{2^s − 1}])` to length
16, built once at init.

**Regime B stage 0 (d=16, 2d=32):** single pair `(V0, V1)`, `p = 0`, so
`base_tw[0] = [1]` and every pair's twiddle vector is `seed[0] =
[W_{32}^0, W_{32}^1, ..., W_{32}^{15}]`. Complex vec-vec butterfly.

Store `V0, V1` back.
