VECTOR REDUCTION OPERATIONS
===========================

Spec Reference: riscv-isa-manual/src/v-st-ext.adoc
Lines: 3829-4035

Overview
--------
Vector reduction operations reduce a vector register group of elements using a
binary operator, combining them with a scalar held in element 0 of a vector
register to produce a scalar result in element 0 of a vector register.

The scalar input and output are in element 0 of a single vector register (not
a vector register group), allowing any vector register to be the scalar
source/destination regardless of LMUL.

Key behaviors:
- Inactive elements are excluded from reduction
- Scalar operand is always included regardless of mask
- vl=0 performs no operation, destination not updated
- vstart must be 0 (raises illegal instruction exception otherwise)
- Destination can overlap source operands including mask register
- Assembler syntax: .vs suffix (vector register group and scalar)

Implementation Summary
----------------------
Currently implemented: 14 instructions (decoded, instruction classes, and execution
via tree reduction decomposition in oamlet/reduction.py per docs/PLAN_reductions.md).
Covers all single-width integer, single-width FP (unordered), widening integer, and
widening FP (unordered) reductions.
Total in spec: 18 reduction instructions
Not yet implemented: ordered FP reductions (vfredosum, vfwredosum)
Coverage: ~78%

VFREDOSUM.VS (Status: ✗)
  Spec: v-st-ext.adoc lines 3924, 3932-3955
  Encoding: funct6=0x03, funct3=OPFVV (0x1)
  Format: vfredosum.vs vd, vs2, vs1, vm
  Implementation: Not implemented
  Description: Ordered FP sum reduction
  Notes:
    - Must sum in element order: vd[0] = (((vs1[0]+vs2[0])+vs2[1])+...)+vs2[vl-1]
    - Each addition operates like scalar FP (exceptions, special values)
    - Supports compiler auto-vectorization
    - Masked-off elements don't affect result or exception flags
    - vl=0: scalar copied without canonicalizing NaN or setting flags

IMPLEMENTATION INFRASTRUCTURE
==============================

Existing Support:
- Vreduction class in instructions/vector.py (handles all 14 decoded reduction ops)
- VRedOp enum in kinstructions.py with all 14 reduction operations
- Decoding for all 14 ops in decode.py (integer, widening integer, FP, widening FP)
- oamlet/reduction.py: tree reduction decomposition into existing kinstrs
  (VArithVv/Vx, VCmpVi, VRgather, Vid, etc.)
- _VREDOP_TO_ARITHOP mapping for all 14 ops
- Widening ops handled via _WIDENING_OPS set

What's Missing:
- Ordered FP reductions (vfredosum.vs, vfwredosum.vs) — require sequential summation
- Comprehensive test coverage beyond dotprod

Test Coverage
-------------
- vredsum.vs tested in tests/dotprod/ (dot product uses reduction)
- No tests for: max/min, bitwise, widening, or any FP reductions

Summary
-------
14 of 18 reduction instructions are implemented via tree reduction decomposition:
- ✓ All 8 single-width integer reductions
- ✓ 2 widening integer reductions
- ✓ 3 single-width FP reductions (unordered sum, max, min)
- ✓ 1 widening FP reduction (unordered sum)
- ✗ 2 ordered FP reductions (vfredosum.vs, vfwredosum.vs)

The tree reduction approach decomposes reductions into existing kinstrs
(VArithVv/Vx, VCmpVi, VRgather, Vid, etc.) — see docs/PLAN_reductions.md.

Note: The synchronization network (synchronization.py) supports MIN value
aggregation across kamlets. This could potentially be used for min/max reductions
(vredmin, vredmax, vfredmin, vfredmax) more efficiently than the current tree
decomposition approach, since the sync network already provides cross-kamlet
reduction. Worth investigating whether this would be faster or simpler than the
current kinstr-based tree reduction for these ops.

Priority for remaining work:
1. Ordered FP reductions - needed for auto-vectorization compliance
2. Comprehensive test coverage
3. Investigate sync network for min/max reductions
