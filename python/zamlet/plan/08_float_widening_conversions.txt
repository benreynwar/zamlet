Overview
--------
Widening FP operations take SEW-bit operands and produce 2*SEW-bit results,
essential for avoiding loss of precision in intermediate computations.

Conversion operations transform between floating-point and integer types,
supporting same-width, widening, and narrowing conversions.

Implementation Summary
----------------------
Currently implemented: 16 widening FP arithmetic instructions
  (vfwadd/vfwsub .vv/.vf and .wv/.wf, vfwmul .vv/.vf, vfwmacc/vfwnmacc/
   vfwmsac/vfwnmsac .vv/.vf)
Total in spec: ~36 widening/conversion instructions
Coverage: ~44%

Implementation: widening FP arithmetic shares the VArithVvOv / VArithVxOv
classes used for the integer widening family (see
06_integer_widening_narrowing.txt for the lowering chain). The
kinstruction's `_ov_arith_fmt` helper picks the float struct format ('d',
'f', 'e') based on the op-category check (op in FLOAT_OPS), so the same
shape/width machinery applies. Float ops set src{1,2}_signed=False at the
wrapper layer; signedness is irrelevant for float fmt selection. Accumulator
ops (FMACC/FNMACC/FMSAC/FNMSAC) read vd as the addend at 2*SEW.

Conversion ops (vf{c,wc,nc}vt*, including vfwcvt.f.f.v and vfncvt.f.f.w)
are still unimplemented — these stay on the unary (Ov-unary) path and are
tracked separately.

WIDENING FP ADD/SUBTRACT (Status: ✓)
=====================================

Spec: v-st-ext.adoc lines 3287-3301

These come in two forms:
1. SEW + SEW → 2*SEW (both operands narrow)
2. 2*SEW + SEW → 2*SEW (one operand wide, .w suffix)

WIDENING FP FUSED MULTIPLY-ADD (Status: ✓)
===========================================

Spec: v-st-ext.adoc lines 3371-3393

All widening FMA operations overwrite the wide addend (vd). The multiplier
inputs are SEW wide, while the addend and destination are 2*SEW bits wide.
Implemented via VArithVvOv / VArithVxOv with shape=BASE; vd is treated as
the accumulator (ACCUM_OPS path) and read at 2*SEW.

REGISTER CONSTRAINTS
====================

Widening Operations:
  - Destination vd has EEW=2*SEW, EMUL=2*LMUL
  - Source operands (for SEW+SEW→2*SEW) have EEW=SEW, EMUL=LMUL
  - For 2*SEW+SEW→2*SEW (.w forms), first source is 2*LMUL
  - vd cannot overlap source register groups

Narrowing Operations:
  - Source vs2 has EEW=2*SEW, EMUL=2*LMUL
  - Destination vd has EEW=SEW, EMUL=LMUL
  - vd cannot overlap vs2

Same-Width Conversions:
  - Both source and destination have EEW=SEW, EMUL=LMUL
  - Standard register group rules apply

Summary
-------
Widening FP add/sub (BASE + WIDE), widening FP multiply, and widening FP
fused multiply-add (FMACC, FNMACC, FMSAC, FNMSAC, .vv and .vf forms) are
implemented through VArithVvOv / VArithVxOv with op set to the float
VArithOp value. The shared Ov machinery handles the per-operand width
unpack/pack.

Implemented (this round): 16 instructions
  - 8 widening FP add/subtract variants (BASE + WIDE shapes, .vv/.vf)
  - 2 widening FP multiply variants (vfwmul .vv/.vf)
  - 8 widening FP fused multiply-add variants
    (vfwmacc, vfwnmacc, vfwmsac, vfwnmsac, .vv/.vf)

Still unimplemented: ~21 conversion instructions
  - 6 single-width FP/integer
  - 7 widening FP/integer
  - 8 narrowing FP/integer (including ROD)

Priority: medium for FP→int/int→FP conversions (commonly emitted by
compilers); FFT path doesn't depend on them.
